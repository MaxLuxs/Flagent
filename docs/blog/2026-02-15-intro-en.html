Why I built Flagent: feature flags that actually fit

15 Feb 2026 · Русский

Hi. I’m Maxim, and I’ve been building Android apps for about five years. Over time I kept running into the same problem: I wanted feature flags, gradual rollouts, and a way to experiment without betting the whole release on one big deploy — but nothing I tried really fit.

Either the stack didn’t match (no proper Kotlin-native story), or it was too heavy for what I needed, or it was yet another SaaS I had to wire into our pipeline. So at some point I thought: why not build something that fits the way I work?

What I was missing

I wanted:

  • Feature flags — turn things on and off without a new app release.
  • Experimentation — A/B tests and rollouts with clear rules, not ad‑hoc ifs in code.
  • Faster, safer releases — ship often, hide behind flags, and kill a feature in one click if something goes wrong.
  • Something that plays nice with Kotlin and the rest of my stack, not a bolt-on from another ecosystem.

I looked at what’s out there and took ideas from several systems, but I couldn’t find one that was “this is exactly what I need.” So I started a pet project. That project is Flagent.

What Flagent is today

It’s an open-source feature-flag and experimentation backend plus a small admin UI. You can self-host it and plug it into your apps. Right now it already has:

  • Backend — Ktor, Clean Architecture, PostgreSQL / MySQL / SQLite.
  • Admin UI — Compose for Web: flags, segments, variants, constraints, a debug console.
  • SDKs — Kotlin (including an “enhanced” one with offline eval and real-time updates), JavaScript/TS, Swift, Python, Go. Plus a Ktor plugin for server-side evaluation.
  • Evaluation — deterministic bucketing (MurmurHash3), targeting by segments and constraints.
  • Client-side evaluation in Kotlin and Go — evaluate locally, no round-trip for every check.
  • Real-time updates (SSE) so clients get new flag state without polling.

So it’s already usable: you can run the server, create flags, connect an SDK, and start gating features or running simple experiments. I use ideas from other platforms, but the design and implementation are tuned for what I care about: type safety, coroutines, and something I can host myself.

How to use it

Get the server up:

docker run -p 18000:18000 ghcr.io/maxluxs/flagent

Open http://localhost:18000, log in (default admin@local / admin), create a flag, grab an API key. Then in your app you add the SDK and evaluate:

if (client.isEnabled("my-feature")) {
    // new behaviour
} else {
    // old behaviour
}

For the full walkthrough, see the Quick Start and the docs. There are guides for configuration, deployment, SDKs, and the API.

Where it’s going

Right now it’s a side project. I’m building it for my own needs first, but I’d like it to grow into something others can rely on too — better docs, more integrations, maybe a managed option later. I’m not promising a fixed roadmap; I’m just sharing what exists and where I’d like to take it.

If this resonates — you’re on Kotlin or Ktor, or you just want a simple self-hosted flag server — I’d really appreciate your support: try it, star the repo, open an issue or a discussion, or share how you’d use it. That feedback is what will shape the next steps.

Thanks for reading. You can find the project on GitHub and more in the documentation.
